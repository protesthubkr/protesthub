# ProtestHub UI Principles

이 문서는 ProtestHub의 공개 화면과 내부 검수 화면을 수정할 때 유지해야 할 UI 원칙을 정리한다. 새 기능을 붙일 때는 먼저 이 문서를 보고, 기존 화면의 정보 구조를 깨지 않는 방향으로 개선한다.

## 제품 성격

ProtestHub는 집회 참여자가 모바일에서 빠르게 일정을 훑고 상세 정보를 확인하는 도구다. 첫 화면은 설명형 랜딩 페이지가 아니라 바로 쓰는 조회 화면이어야 한다.

- 공개 화면의 우선순위는 날짜, 시간, 제목, 장소, 주최, 의제 태그다.
- 원본 출처와 포스터 확대는 상세 페이지에서 다룬다.
- 지도, 로그인, 개인화, 추천 묶음은 MVP의 핵심이 아니다.
- 내부 검수 화면은 예쁜 홍보 화면이 아니라 빠르게 분류하고 공개하는 작업 화면이다.

## 설계 원칙

### 1. 스캔 우선

목록과 캘린더는 사용자가 모든 문장을 읽는 화면이 아니다. 눈으로 훑었을 때 날짜별 밀도, 시간대, 장소, 제목이 먼저 보여야 한다.

- 목록 카드는 한 집회의 핵심만 담는다.
- 캘린더 셀은 날짜별 진입점이다. 긴 제목을 모두 보여주려고 셀을 복잡하게 만들지 않는다.
- 같은 정보가 여러 곳에서 반복되면 한 곳을 줄인다.

### 2. 필터는 보조 장치

필터는 사용자가 원하는 일정을 빨리 좁히기 위한 보조 장치다. 필터를 선택하지 않아도 전체 목록을 볼 수 있어야 한다.

- 빈 필터 배열은 전체 조회다.
- 적용된 조건은 상단 칩에 노출한다.
- 필터는 한 번에 적용하는 batch interaction을 기본으로 한다.
- 개별 조건은 칩 또는 필터 패널에서 쉽게 제거할 수 있어야 한다.

### 3. 상태는 명시한다

로딩, 빈 결과, 오류, 비활성, 과거 날짜, 현재 날짜, 검수 상태는 서로 다른 상태다. 서로 비슷한 문구나 같은 모양으로 뭉개지 않도록 한다.

- 로딩 중에는 “없음” 화면을 보여주지 않는다.
- 오류는 조용히 사라지게 하지 않고 다시 시도할 단서를 둔다.
- 과거 날짜는 비활성 색과 disabled 동작을 함께 적용한다.
- 관리자 상태 탭은 현재 상태와 개수를 동시에 보여준다.

### 4. 모바일 터치 기준

주요 조작 대상은 최소 44px 높이를 기본으로 한다. 작은 텍스트 링크가 필요하면 주변 여백을 포함한 터치 영역을 확보한다.

- 상단 아이콘 버튼, 필터 버튼, 하단 시트 버튼, 월 이동 버튼은 44px 이상이다.
- 아이콘만 있는 버튼은 `aria-label`과 분명한 시각 상태를 가진다.
- 서로 가까운 조작 요소 사이에는 충분한 간격을 둔다.

### 5. 위계는 적게, 일관되게

모든 텍스트를 굵게 만들면 아무것도 강조되지 않는다. 제목, 메타, 보조 설명, 상태 문구의 굵기와 크기를 구분한다.

- 제목: 굵게, 줄 수 제한은 화면 맥락에 맞게.
- 메타 라벨: 작고 차분하게.
- 날짜/시간: 목록 구조를 잡는 정보이므로 카드 제목과 경쟁하지 않게 한다.
- 버튼 텍스트: 화면 전체보다 강하지 않게, 행동의 중요도만 전달한다.

### 6. 색상은 의미를 가진다

의제 색상은 대표 태그와 연결되는 정보다. 장식용으로 색을 남발하지 않는다.

- 카드 왼쪽 accent와 의제 badge는 같은 의미 체계를 쓴다.
- 선택 상태는 배경과 border가 함께 바뀐다.
- 비활성 상태는 색만이 아니라 커서, opacity, disabled 속성을 함께 쓴다.
- 색상만으로 의미를 전달하지 않는다.

## 화면별 기준

### 공개 목록

- 상단 조건 칩은 현재 조회 조건을 보여주는 영역이다.
- 뷰 전환 버튼은 현재 뷰가 아니라 이동할 뷰를 나타내므로 `aria-label`을 명확히 둔다.
- 날짜 헤더는 날짜와 건수를 좌우로 나눠 빠르게 읽히게 한다.
- 시간 라벨은 텍스트와 구분선으로 처리하고, 같은 시간대 카드는 그 아래로 묶는다.
- 카드 전체가 상세 페이지 링크라는 점이 보여야 한다.
- 이전 일정 pull-to-load는 긴 문장 배너가 아니라 중앙 원형 진행 indicator와 짧은 상태 라벨로만 피드백한다.

### 캘린더

- 월 캘린더는 full-width grid로 둔다.
- 셀은 날짜, 최대 소수의 제목 샘플, overflow count만 담는다.
- 오늘 이전 날짜는 비활성화하고 클릭되지 않게 한다.
- 오늘 날짜는 선택 상태가 아니라 현재 날짜 표시만 한다.
- 특정 날짜 클릭 시 리스트 뷰로 이동하고, 그 날짜부터 이후 일정이 이어져야 한다.

### 필터 시트

- 하단 시트는 모바일에서 가장 중요한 필터 진입점이다.
- 닫기와 적용은 시트 상단에 두고, 탭은 하단에 고정한다.
- 선택 항목은 버튼형 선택지로 처리한다.
- 의제와 지역은 2열, 주최는 1열을 기본으로 한다.
- “전체”는 실제 전체 선택/해제를 수행하지만, URL에는 전체 조건을 남기지 않는다.

### 상세 페이지

- 상세 페이지는 사용자가 참여 가능성을 판단하는 화면이다.
- 제목과 태그 다음에 일시, 장소, 상세장소, 출처를 먼저 보여준다.
- 포스터가 없으면 빈 포스터 영역을 만들지 않는다.
- 원본 포스트 링크는 포스터 아래 또는 포스터가 없을 때 상세 섹션 하단의 안정적인 CTA로 둔다.
- 취소 페이지는 공개 목록과 다른 상태임을 첫 화면에서 분명히 알린다.

### 내부 검수

- 검수 화면은 작업 효율이 우선이다.
- 상태 탭, 범위 탭, 수동 추가, 후보 카드는 서로 다른 작업 단위로 보이게 한다.
- 후보 카드에서는 상태 변경, 원본 보기, 구조화, 공개 폼이 시각적으로 섞이지 않게 한다.
- X 상세 수집처럼 API 비용이 드는 작업은 기본 수집과 분리하고, 후보를 확인한 뒤 실행하는 보조 액션으로 둔다.
- 버튼 색은 중요도와 위험도에 맞춰 구분한다.

## 구현 체크리스트

- 새 UI 요소는 기존 CSS 토큰을 먼저 사용한다.
- 반복되는 버튼/칩/상태 블록은 같은 class family를 쓴다.
- `src/app` route에 UI 상태 계산을 추가하지 않는다.
- 공개 목록 API payload에 주소 상세, 포스터를 다시 싣지 않는다.
- 프론트엔드 변경 후 모바일 390x844에서 목록, 캘린더, 필터, 상세을 확인한다.
